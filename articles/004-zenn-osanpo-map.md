# 「最短じゃなくていい」散歩アプリを、月3ドルでつくる

Googleマップを開けば、最短・最速の道はすぐ出てくる。
でも散歩でほしいのは、むしろその逆だ。

少し寄り道して、公園や商店街や小さなカフェに触れてからゴールに着く——そんなルートを提案する Web アプリを、個人開発でつくっている。名前は「お散歩マップ」。

この記事では、実装の細部よりも、**最初に決めておいた設計の芯**を残しておく。

---

## プロダクトの一言

スタート・ゴール・所要時間を入れると、好みに合う見どころを挟んだ散歩ルートを2〜3本返す。

対象はまず東京23区と川崎・横浜。好みは「散歩診断」でもよいし、スキップして「自然 / 食べ歩き / お買い物」の3択でもよい。どちらも同じエンジンに流す。

やらないことも最初に決めた。共同プラン、歴史テーマ、全国展開は後回し。MVPは「一人で、今日の散歩を決められる」まで。

---

## 設計を縛ったもの: 月3ドル

技術選定の前に、予算を固定した。**月額およそ3ドル**。

この数字があると、迷う選択肢がかなり減る。

- Google Maps / Places は使わない
- 常時起動のサーバは置かない
- AIは「毎回フル生成」しない

代わりに、無料またはほぼ無料で組める部品を選ぶ。

| 役割 | 採用 |
|------|------|
| フロント | React + TypeScript |
| 地図表示 | MapLibre + 無料タイル |
| 地名検索 | Nominatim（OSM） |
| 徒歩の線 | OSRM / OpenRouteService |
| 見どころの文章 | Bedrock（安いモデル） |
| API / DB | Lambda + DynamoDB |
| 配信 | S3 + CloudFront |

「高機能な地図APIを買わない」と決めた瞬間に、アプリの顔もはっきりした。最短ナビの模倣ではなく、散歩専用のシンプルな画面でよい、と。

---

## 判断1: フロントは React（あとで一緒に触れる想定）

MVPは一人用だ。それでもフロントに React を選んだのは、将来の画面の形を先に決めておきたかったから。

ロードマップには、複数人で同じ散歩プランを作り、好みを折衷する共同ルートがある。その先には、マップ上の見どころや経由点を**みんなでリアルタイムに編集する**体験も想定している。地図とサイドパネルが同時に動き、誰かの操作がすぐ画面に乗る——そういう UI は、状態とコンポーネントの境界がはっきりしている方が運びやすい。

いまはまだ同期も共同編集もない。地図＋パネルの操作感を React で組み立てておき、あとから共同編集の層を足せるようにした、という順番だ。

---

## 判断2: 「通る場所」と「線の形」を分ける

最初のプロトタイプでは、スタート・見どころ・ゴールを直線で結んでいた。見た目は地図アプリなのに、道を無視して空を飛んでいる感じになる。

ここでやりがちなのが、タイル画像から道路をトレースするような凝った処理だ。精度も保守も重いので、MVPでは捨てた。

代わりに役割を分けた。

1. **waypoints** … どこを通るか（論理）
2. **徒歩ルーティング** … 道に沿った折れ線（見た目）
3. **MapLibre** … その折れ線をそのまま描く

アプリ側は「きれいな線を自分で計算する」必要がない。外部の徒歩ルートAPIに渡して、帰ってきた座標列を描くだけにする。APIが落ちたときだけ、直線にフォールバックする。

散歩アプリの肝は、ここだと感じている。**楽しさはwaypointsで決め、納得感はgeometryで担保する。**

---

## 判断3: AIに店名をInventさせない

見どころの紹介文や並びはLLMに任せたい。一方で、存在しない店を自信満々に出されると、散歩アプリとしては致命傷になる。

なので生成は2段にした。

```
OSM（Overpass）で実在POIを取る
    ↓
Bedrockで選別・並べ替え・紹介文
    ↓
ルートに組み込む
```

LLMの仕事は「ゼロから場所を考える」ことではない。すでに存在する候補の中から、テーマに合うものを選び、散歩者向けの短い文を書くことだ。

コスト面でも同じ方向を向いている。同じエリア・同じ好みの再リクエストは、DynamoDBのキャッシュから返す。Bedrockはミスったときだけ。月3ドルの前提では、キャッシュはオプションではなく必須条件に近い。

---

## 判断4: 地図まわりは全部プロキシする

地名検索も地図クリック時の逆ジオコーディングも、ブラウザから無料APIを直叩きしたくなる。実際にはCORSや利用規約、レート制限があるので、Lambda側に薄いプロキシを置いた。

ユーザーから見ると「検索して選ぶ」か「地図をタップする」かの違いだけ。内部ではどちらも同じ地点データに正規化する。UIは二つ、データモデルは一つ、というだけの話だが、ここを雑にするとあとで必ず二重管理になる。

---

## 判断5: 配信は Amplify ではなく CloudFront

フロントの置き場所は、最初から Amplify Hosting も候補だった。git push でビルド・配信まで面倒を見てくれるし、個人開発ではいちばん楽な道だ。

それでも **S3 + CloudFront を CDK で明示する**方を選んだ。いちばん正直な動機から書くと、**もともと立てている別サイトが Amplify 採用**で、配信まわりの感覚はそちらで足りている。だからこのプロジェクトでは、Amplify の裏で動いているのと同系統の CDN を、**自分で組み立てる側から見てみたかった**。速さや「CDNが強いから」ではない。

そのうえで、お散歩マップ向きでもあった理由が次だ。

**1. アプリ全体を一枚の設計図に載せたかった**

本番は静的フロントだけではない。API Gateway、Lambda、DynamoDB、Bedrock の IAM まで含めて初めて動く。Amplify にフロントだけ預け、API は別管理、という二系統にすると、デプロイ手順も権限も環境変数も二重になる。既存サイト側で Amplify の楽さは知っているので、こちらは CDK でフロント配信から API・DB・AI 権限まで `infra/` に並べる実験にした。月3ドルで部品を増やしすぎないプロジェクトほど、入口は一つにしたい。

**2. コストの見え方を自分で縛りたかった**

Amplify は便利なぶん、何がいくらなのかがぼやけやすい。こちらは最初から「S3 と CloudFront はだいたいこれくらい」と予算表に載せたうえで組んでいる。請求の単位が設計図と同じ語彙で読めると、Bedrock のキャッシュや Lambda の呼び出しと並べて判断しやすい。ホスティングに魔法を期待するより、安い部品を明示した方が、この予算感には合う。

**3. このアプリに Amplify の便利さが、まだ必要ではなかった**

ブランチごとのプレビュー URL や Console 上の環境変数は魅力的だ。ただ MVP のフロントは Vite でビルドした SPA を配るだけなので、git 連携のビルド基盤がなくても困らない。地図の操作感はホスティング方式と無関係で、MapLibre はブラウザと API さえあれば動く。あとから CI で `build` → `deploy` を自動化すればよく、最初から Amplify に寄せる必然は薄かった。

対比だけ書くと、だいたいこうなる。

| | Amplify Hosting | お散歩マップ（CDK + CloudFront） |
|--|--|--|
| デプロイ | git push で自動ビルド | 手元または CI で `build` → `cdk deploy` |
| インフラ | 裏で S3/CloudFront 等が立つ | `infra/` に全部書いてある |
| フロントと API | 分かれやすい | 同じスタックに同居 |
| コストの見え方 | ホスティング単位でまとまりやすい | S3 / CloudFront / Lambda を個別に読める |
| CDN | CloudFront 系統 | 同じ系統 |

覚え方はシンプルだ。Amplify = ホスティングが面倒を見る（既存サイトはこちら）。CDK + CloudFront = 設計図を自分で書いて、配信も含めて同じ帳面に残す。慣れた道を一度やめて、組み立て側を試した、ということでもある。

---

## キャッシュを「1日鮮度」にする

キャッシュは必須だが、永久保存だと OSM 上の店の開閉が反映されない。かといって、毎日すべてのキーを Bedrock で作り直すと月3ドルが溶ける。

なので鮮度はこう付ける。

- 書き込みから **24時間** でソフト期限切れ
- 期限切れでも定時ジョブは回さない
- **次にそのエリアがリクエストされたときだけ** OSM + Bedrock で更新
- 誰も触らない古いキーは DynamoDB TTL で消す（掃除用 Lambda は置かない）

「1日に1回更新したい」気持ちは、先回り再生成ではなく、**鮮度の上限を1日にする**ことで満たす。利用があるキーだけ課金されるので、予算の向きと合う。

---

## あえて後回しにしたもの

認証はいま自前JWTだが、本番らしい仕組みはPhase 2でCognito Liteに移す前提で書いてある。個人開発の規模なら無料枠に収まりやすいし、最初からAuthに時間を溶かすより、ルートの質を上げた方がプロダクトらしい。

同様に、共同お散歩やリアルタイム編集も面白いが、一人用が動いてからの話にした。フロントの土台だけ先に React で置き、同期や折衷ロジックは後から足す。先に欲張ると、画面もデータも一気に増える。

---

## いまの感触

「最短を出さない」と言うのは簡単で、実際にやるとうまくいく条件が狭い。

- 見どころが実在すること
- 線が道に乗っていること
- 同じ条件で何度も課金しないこと
- Googleマップのコピーに見えないこと

月3ドルという制約は厳しいが、おかげで構成が素直になった。無料の地図データ、薄いAPI、必要なときだけAI、あとはキャッシュ。フロントは共同編集を見据えて React、配信は Amplify に任せきりにせず CloudFront を設計図に残した。派手さはないが、散歩アプリにはちょうどいい厚みだと思う。

次はソフトTTLを本番に載せつつ、ルート生成の安定化と保存・認証まわりを少しずつ寄せていく。同じエリアを歩いても「今日はこっち」と思える候補が、古くなりすぎず安定して出るようにしたい。
